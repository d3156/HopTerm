# UI_FLOW — Пользовательские сценарии и экраны

Документ описывает GUI по мокапу `index.html` и привязывает каждый экран/флоу к
конкретным view и требованиям ТЗ §9 (UX) и §5.4 (GUI). Реализация — крейт `ui`
(iced 0.13), который оперирует состоянием/командами `app` и типами `domain`.

---

## 1. Карта экранов

Мокап `index.html` задаёт каркас: **sidebar слева** + **main area** с четырьмя
view, переключаемыми навигацией. В коде это `enum View { Hosts, Commands,
Transfers, Settings }` в состоянии `ui` (аналог `showView()` в мокапе).

```
┌─────────────┬───────────────────────────────────────────────┐
│  SIDEBAR    │  TOPBAR (заголовок + быстрые действия)         │
│             ├───────────────────────────────────────────────┤
│ HopTerm 🌓  │                                               │
│             │                                               │
│ Навигация   │            CONTENT (активный view)            │
│ • Хосты  4  │                                               │
│ • Команды 7 │   Hosts / Commands / Transfers / Settings     │
│ • Передачи 2│                                               │
│ • Настройки │                                               │
│             │                                               │
│ Быстрый     │                                               │
│ доступ      │                                               │
│ ● prod-..   │                                               │
│ ● k8s-..    │                                               │
│ ○ dev-..    ├───────────────────────────────────────────────┤
│ ○ bastion   │  STATUS BAR (состояние соединения)            │
│ v0.1.0-alpha│                                               │
└─────────────┴───────────────────────────────────────────────┘
```

### Sidebar (постоянный)
- **Header:** логотип HopTerm + переключатель темы (`theme-toggle` → §5.4
  «настройки темы», Settings).
- **Навигация:** Хосты / Команды / Передачи / Настройки, с бейджами-счётчиками.
  Активный пункт подсвечен (`nav-item.active`).
- **Быстрый доступ:** список хостов со `status-dot` online/offline — это
  «менеджер подключений» / quick switcher (§5.4, §9.1). Клик = быстрое
  переподключение по профилю (§4.5).
- **Footer:** версия + стек.

### Status bar (§5.4 — индикатор состояния)
Отражает `ActiveSession.connection_state`: `Connecting{hop_index}` →
«Подключение… hop 2/3», `Connected` → имя target, `Reconnecting{attempt}`,
`Failed{hop_index}`. Источник — поток статусов из `app` (ARCHITECTURE §6).

## 2. VIEW: Hosts (`#view-hosts`) — менеджер подключений

Главный экран (§9.1). Topbar: заголовок «Хосты», кнопка **Поиск** (§5.4
желательное «поиск по сохранённым подключениям») и **Добавить хост** (открывает
add-host modal).

- **Tabs-фильтры:** «Все хосты / Production / Dev-Staging / Bastion» — фильтр по
  `HostProfile.labels`.
- **Host grid:** карточки хостов (`HostProfile`). На карточке:
  - имя + `username@address:port` (моноширинно);
  - бейдж статуса (online/offline);
  - meta: число хопов («2 хопа» / «прямой»), число привязанных команд, бейдж
    типа auth (`ed25519` / `password` / `rsa-4096`);
  - hover-actions: редактировать, дублировать.
- **Действия:** клик по карточке → открыть/редактировать профиль (add-host
  modal). Быстрые действия Connect / New Session / Reconnect (§9.1) — из
  topbar/контекста карточки, команды уходят в `app`.

Привязка к §9.1: «список/дерево подключений слева» = sidebar + этот grid;
быстрые действия = topbar.

## 3. Флоу: создание multi-hop подключения (add-host modal)

Модалка `#modal-add-host` — ядро сценария §9.2/§9.3 и §5.2. Открывается из
кнопки «Добавить хост» или клика по host-card.

### Структура модалки
1. **Имя подключения** (`name`) — отображается в sidebar и host-grid.
2. **Цепочка хопов** (`hop-list`) + кнопка **«Добавить хоп»** (`addHop()`):
   - Каждый `hop-item` = один `Hop`: бейдж с номером, поля
     **Пользователь / Порт / Хост-IP**, селектор **Аутентификация**
     (SSH ключ / Пароль / SSH Agent = `AuthMethod`) и поле
     **Пароль / Путь к ключу**.
   - **Кнопка удаления хопа** (`removeHop`) — менять состав цепочки.
   - **Целевой хост** — последний элемент, выделен цветом primary, бейдж 🎯,
     `is_target=true`. Удалить нельзя.
   - Note-box поясняет семантику: «Хопы выполняются последовательно. Последний —
     целевой. Каждый предыдущий — jump-сервер (ProxyJump)» (§5.2).
3. **Breadcrumb маршрута** (`hop-chain`) — живой предпросмотр §9.3:
   ```
   Маршрут: localhost → admin@bastion.example.com:22
                      → ubuntu@10.10.0.5:2222
                      → admin@192.168.1.100:22   (final, primary)
   ```
   Обновляется при каждом изменении полей/состава цепочки.
4. **Footer:** **Отмена** / **Тест соединения** / **Сохранить**.
   - «Тест соединения» (§9.2 «тестировать маршрут») → `app` строит цепочку и
     эмитит `RouteProgress`/`RouteError` по hop'ам, не сохраняя профиль.
   - «Сохранить» → валидирует и пишет `HostProfile`+`JumpRoute`+`SessionProfile`
     через `ProfileStore` в `config.toml`.

### Изменение порядка хопов (§9.2)
Порядок (`Hop.order`) задаётся позицией в `hop-list`; добавление/удаление и
(после MVP — drag) перестановка обновляют breadcrumb и сохраняемую цепочку.

### Маппинг на состояние `ui`
```
AddHostForm {
  name: String,
  hops: Vec<HopForm>,        // последний помечен is_target
}
HopForm { username, port, host, auth_kind, secret_or_keypath }
Messages: AddHop, RemoveHop(idx), HopFieldChanged(idx, field, val),
          TestRoute, SaveHost, CloseModal
```

## 4. VIEW: Commands (`#view-commands`) — быстрые команды

Сохранённые операции, привязанные к хосту (download / upload / exec / pipeline).
Topbar: фильтр по хосту + **«Новая команда»** (`#modal-add-command`).

- **cmd-card:** бейдж типа (СКАЧАТЬ/СКРИПТ/ЗАГРУЗИТЬ), имя, хост-тег, превью
  команды (моноширинная подсветка шагов), бейджи-параметры, кнопка **▶ Запустить**.
- **Конструктор команды** (`#modal-add-command`): название + хост + тип операции
  (↓ Скачать / ↑ Загрузить / ▶ Выполнить / ⛓ Пайплайн), пошаговый builder
  (например для download: исходная папка → сжатие tar.gz/bz2/zst → локальное
  назначение + распаковка), **живой предпросмотр** итоговой команды и тумблер
  «Требовать подтверждения перед запуском».

Связь с ТЗ: это надстройка над transfer/exec поверх той же цепочки; «Подтверждение
перед exec-командами» = тумблер из Settings (§6.3 — деструктивные операции). Эта
view выходит за обязательный MVP и относится к polish/расширяемости (§6.4), но
переиспользует `TransferService` и shell-канал.

## 5. VIEW: Transfers (`#view-transfers`) — передача файлов (§9.4, §5.5)

Отдельная панель передач. Topbar: «Активные передачи» + «Очистить завершённые».

- **transfer-block на каждый `TransferJob`:**
  - заголовок (имя задачи) + статус (Выполняется/Загрузка с `pulse`-индикатором,
    Done) + кнопка **Отмена** (§5.5 «отмена операции»);
  - **progress-item на каждый файл:** имя (моно), `X МБ / Y МБ`, progress-bar;
    завершённые — зелёные (`done`);
  - **transfer-stats:** Скорость (МБ/с), Осталось (ETA), Передано, Источник/
    Назначение (хост), Протокол (SFTP).
- Прогресс приходит потоком из `transfer` через subscription (ARCHITECTURE §5);
  UI не блокируется (§6.1).

### UX выбора файла
Upload/Download инициируются из host-card или Transfers через нативный диалог
`rfd` (§5.5). Конфликты overwrite/rename/skip (`TransferPrefs.conflict_policy`)
показываются модальным выбором. Журнал завершённых/ошибочных операций (§9.4) —
блоки со статусом Done/Failed; «Очистить завершённые» убирает их.

§9.4 «локальная и удалённая панели / remote browser» — remote directory listing
(через `TransferService::list_dir`) добавляется как панель навигации; в MVP
достаточно выбора путей и панели прогресса.

## 6. VIEW: Settings (`#view-settings`) — настройки (§10)

Карточки:
- **Конфигурация** — тумблеры: «Сохранять конфиг на диск» (`config.toml`),
  «Хранить пароли хопов» (keychain/зашифрованный файл — §6.3), «Сжимать перед
  скачиванием» (tar -czf по умолчанию), «Подтверждение перед exec-командами».
- **Путь к конфигу** — `~/.hopterm/` (файлы: `config.toml`, `known_hosts`,
  `keys/`).
- Переключатель темы — в sidebar header (§5.4 «настройки темы»).

Маппится на `[settings]`/`[ui]` секции `config.toml` (см. DATA_MODEL §5).

## 7. Host-key confirmation dialog (§5.4, §5.1)

Отдельный модальный диалог, **не из мокапа напрямую**, но обязательный по §5.4
(«диалог host key confirmation») и §5.1 (fingerprint при первом подключении,
TOFU/явное подтверждение). Триггерится событием `TrustPrompt` из `ssh`/`security`
во время построения цепочки.

```
┌──────────────────────────────────────────────┐
│  Неизвестный ключ хоста                       │
│  Хост:  192.168.1.100:22  (hop 3 / target)    │
│  Алгоритм:  ssh-ed25519                       │
│  Отпечаток: SHA256:Zm9vYmFyYmF6cXV4Li4u...    │
│                                               │
│  Ключ не найден в known_hosts.                │
│  Доверять и продолжить подключение?           │
│                                               │
│   [ Отклонить ]   [ Доверять однократно ]     │
│                   [ Доверять и запомнить ]    │
└──────────────────────────────────────────────┘
```
- «Доверять и запомнить» → запись `HostKey` в `known_hosts` (TrustPolicy::TOFU).
- Указывается, на каком **hop-уровне** возник запрос (важно в multi-hop).
- При изменившемся ключе (mismatch) — предупреждение повышенной строгости.

## 8. Ошибки по hop-уровням (§5.4, §6.2, §11)

Окно ошибок с техническими подробностями (§5.4). Поскольку соединение строится
поэтапно, ошибка несёт `hop_index`, `phase` и `source` (`RouteError`,
DATA_MODEL/TRAITS). UI показывает, **где именно** упало:

```
┌──────────────────────────────────────────────┐
│  Не удалось подключиться                       │
│  Маршрут:                                      │
│   localhost ✔ → bastion.example.com ✔          │
│             → 10.10.0.5:2222 ✘  (hop 2)        │
│             → 192.168.1.100  —                 │
│                                               │
│  Этап: аутентификация                          │
│  Hop 2 (sec-gw, ubuntu@10.10.0.5:2222):        │
│    Authentication failed (publickey)          │
│                                               │
│  [ Подробности ▾ ]   [ Изменить профиль ]      │
│  [ Повторить ]       [ Закрыть ]               │
└──────────────────────────────────────────────┘
```
- Breadcrumb отмечает успешные хопы (✔), упавший (✘) и недостигнутые (—).
- «Подробности» раскрывают технический stack/лог (из `logging`, секреты
  отредактированы — §6.3).
- Статусбар синхронно показывает `Failed{hop_index}`.

---

## 9. Hotkeys и отзывчивость (§5.4, §6.1)

- Hotkeys для основных действий: новый таб, переключение сессий (quick switcher),
  connect/reconnect, upload/download, открыть command palette (после MVP §12).
- Все длительные операции (connect, transfer) — async-команды; `view()`/`update()`
  не блокируются; терминал остаётся отзывчивым при потоке вывода (coalescing
  обновлений экрана, ARCHITECTURE §6).
