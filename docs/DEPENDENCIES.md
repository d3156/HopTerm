# DEPENDENCIES — Зависимости и обоснование

Версии указаны как в `hopterm/Cargo.toml` (`[workspace.dependencies]`).
Сквозное требование ТЗ §3: **избегать C-зависимостей при наличии зрелой
pure-Rust альтернативы**, не использовать Electron / тяжёлый web-stack как
основную GUI-платформу. Ниже — по каждой зависимости: зачем она и какие
альтернативы отвергнуты.

## GUI: `iced` 0.13 — отвергнуты egui и Tauri

Выбран **iced** (рекомендация ТЗ §3). Elm-подобная архитектура
(`Message`/`update`/`view`/`subscription`) идеально ложится на требование §7.2
(GUI отделён от транспорта): UI — чистая функция от состояния, а вся сетевая
работа уходит в команды/подписки на tokio. Feature `tokio` даёт общий runtime,
`advanced`/`lazy` — кастомный виджет терминала и дешёвую перерисовку при потоке
вывода, `image` — иконки.

- **egui отвергнут:** immediate-mode и привязка перерисовки к кадрам хуже
  подходят для «событийного» приложения с долгими async-потоками; интеграция с
  внешним async-стейтом менее естественна, чем подписки iced.
- **Tauri / любой web-stack отвергнут прямо по §3** (нельзя Electron/тяжёлый
  web-stack как основную платформу): тянет webview/JS-рантайм, увеличивает
  surface и стартовое время (§6.1 — «быстро открываться»).
- **GTK/Qt отвергнуты:** тяжёлые C/C++-зависимости, конфликт с §3 (pure-Rust).

## SSH: `russh` 0.61 — отвергнут libssh2-sys

**russh** — pure-Rust async SSH-стек на tokio. Критично для multi-hop (§5.2):
russh даёт программный контроль над открытием `direct-tcpip`-каналов и позволяет
**запускать новый клиентский handshake поверх уже открытого канала** — это и есть
механизм построения цепочки jump-хостов произвольной длины. Ключи (RSA/ed25519,
passphrase-protected) — через `russh::keys` (в 0.61 keys реэкспортируются из
основного крейта, отдельный `russh-keys` не нужен).

- **libssh2-sys / ssh2 отвергнут:** биндинг к C-libssh2 — прямое нарушение §3
  (C-зависимость), плюс блокирующий API, плохо дружащий с async и со сложными
  proxy-цепочками.
- **thrussh отвергнут:** предшественник russh, не развивается; russh — активный
  форк-наследник.
- **Шелл из OpenSSH (`ssh -J`) отвергнут:** внешний процесс лишает контроля над
  host-key prompts, прогрессом и пер-hop диагностикой (§6.2, §11).

## SFTP: `russh-sftp` 2.3 — отвергнут SCP-only и shell-парсинг

**russh-sftp** работает как subsystem поверх готового russh-соединения к target —
значит, передача файлов идёт **через ту же multi-hop цепочку** (§5.2, §5.5), без
отдельного коннекта. Pure-Rust, async, даёт нужные операции: read/write с
чанками (прогресс, §6.1), opendir/listdir, mkdir/rename/remove/stat (§5.5).

- **Парсинг вывода `scp`/`sftp`-бинарей отвергнут:** хрупко, нет надёжного
  прогресса и кодов ошибок. SCP оставлен лишь как *возможный fallback после MVP*
  (§5.5 «желательно»).

## Терминал: `alacritty_terminal` 0.26 — отвергнут «голый» vte

**alacritty_terminal** — это уже **полная модель экрана**: VT/ANSI парсер +
`Grid` + scrollback + обработка resize + Unicode/атрибуты (§5.3). Проверен боем
на `vim`/`tmux`/`htop`/`less` (требование §5.3, критерий §14). Мы используем его
state-машину, а рендерим своими средствами в iced.

- **`vte` напрямую отвергнут:** `vte` — только парсер escape-последовательностей;
  пришлось бы самим писать grid, scrollback, режимы курсора, wrapping — это
  переизобретение ядра alacritty с риском багов на сложных TUI.
- **Свой VT-парсер отвергнут:** объём работы и риск несовместимости несоразмерны.

## Локальный PTY: `portable-pty` 0.9
Для локальных вкладок и fallback-режима (§3): кроссплатформенное открытие PTY
(из экосистемы wezterm). Для удалённых сессий PTY предоставляет сервер через
russh; portable-pty нужен именно локальной части.

## Async: `tokio` 1 (+ `async-trait` 0.1, `futures` 0.3)
**tokio** — рекомендованный рантайм (§3) и тот же, на котором работает iced
(feature `tokio`) и russh; один рантайм на сеть и UI (§6.1 — UI не блокируется).
`async-trait` нужен, потому что domain-трейты (`SshTransport`, `TransferService`,
…) имеют async-методы и используются как `dyn`-объекты. `futures` — для
`Stream`-ов прогресса передач и вывода терминала.

- Альтернативы (`async-std`, `smol`) отвергнуты: russh/iced ориентированы на
  tokio; смешение рантаймов усложняет систему без выгоды.

## Конфиг: `serde` 1 + `toml` 1 — отвергнуты JSON/YAML
ТЗ §10 прямо предлагает TOML как читаемый формат. **serde** — стандарт
сериализации, **toml** — человекочитаемый конфиг (`~/.hopterm/config.toml`).

- **JSON отвергнут:** нет комментариев, менее удобен для ручного редактирования
  инженером (§17 — целевой пользователь правит конфиги руками).
- **YAML отвергнут:** значимые отступы и неоднозначности типов; TOML строже и
  предсказуемее для секций host/hop.

## Логирование: `tracing` 0.1 + `tracing-subscriber` 0.3 — отвергнут `log`
ТЗ §11 требует **structured logs**, уровни severity, трассировку этапов
multi-hop. **tracing** даёт spans (этап `connect hop[i]`, transfer-job как span)
и структурированные поля — это и есть «audit-friendly events» §7.1.

- **`log` отвергнут:** только плоские строковые сообщения, без spans и
  структурных полей — не покрывает §11. Redaction секретов (§6.3) делается на
  уровне нашего `logging`-крейта (типы-обёртки, не реализующие `Debug`/`Display`
  для секретов).

## Пути: `directories` 6
Кроссплатформенное определение home/config-каталога для `~/.hopterm/` (§10).
Pure-Rust, без хардкода `$HOME`.

## File-диалоги: `rfd` 0.17
Нативные диалоги выбора файла/папки для upload/download (§5.5). Pure-Rust обёртка
над системными диалогами, интегрируется с iced/async.

## Идентификаторы: `uuid` 1 (v4, serde)
`id`-поля сущностей (`HostProfile.id`, `JumpRoute.id`, `TransferJob.job_id`, …,
§8). Стабильные уникальные ключи, сериализуемые в TOML.

## Ошибки: `thiserror` 2 (библиотеки) + `anyhow` 1 (app)
ТЗ §16 — «production-minded error handling». **thiserror** — типизированные
доменные ошибки в крейтах-библиотеках (нужны для диагностики по hop-уровням §6.2:
`RouteError{ hop_index, .. }`). **anyhow** — на границе `app`, где удобно
агрегировать разнородные ошибки с контекстом.

## Криптопримитивы: `sha2` 0.11 + `base64` 0.22
Вычисление и отображение **fingerprint host-key** (§5.1, §6.3): SHA-256 от ключа
(`sha2`) → base64 (`base64`) в формате `SHA256:…`, как в OpenSSH. Pure-Rust
(RustCrypto), без C. Сами SSH-ключи разбирает `russh::keys`.

## Сводка соответствия §3 (pure-Rust, без C)

Весь критический путь — pure-Rust: iced, russh, russh-sftp, alacritty_terminal,
portable-pty, tokio, serde/toml, tracing, sha2/base64. C-биндинги (libssh2-sys,
GTK/Qt, webview) сознательно исключены. Это удовлетворяет §3 и упрощает
кроссплатформенную сборку и поставку.
